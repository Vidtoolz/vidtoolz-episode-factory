## 3. The Cockpit Server (package-engine-server.js)

**Size:** 6,488 lines, single file.
**Stack:** Node.js built-in `http` module only. No Express. No framework.
**API endpoints:** 47 defined as named constants at top of file.
**Test coverage:** 803 tests in 21 test files, all passing.

### 3.1 Endpoint groups (all constants defined lines 10–69)

| Group | Prefix | Purpose |
|-------|--------|---------|
| Package Engine | `/api/package-engine/*` | Episode packages, thumbnails, status |
| Package Runs | `/api/package-runs/*` | Per-run artifacts (rough-cut, second-cut, final, export, delivery, pickup-plan) |
| Capture/Evidence | `/api/package-runs/capture-evidence/*`, `/api/package-runs/evidence-intake/*` | Media capture workflow |
| AIGEN Pipeline | `/api/aigen/*` | Production pipeline status, Resolve assembly, FLUX images, selected images |
| PRESTO Batch | `/api/presto/*` | submit, job-status, cancel, results |
| FLUX Batch | `/api/flux/*` | submit, job-status, cancel, results |
| Image Prompts | `/api/image-prompts/*` | read, validate, save |
| Static asset | `/aigen-assets/*` | Image thumbnail serving with traversal guard |

### 3.2 Routing pattern
Manual `if (req.method === 'GET' && url.pathname === X)` chains. First match at line ~5854 in the request handler.

### 3.3 Write protection
Uses a nonce-based header mechanism:
```js
const LOCAL_WRITE_NONCE = crypto.randomBytes(24).toString('hex');
const LOCAL_WRITE_NONCE_HEADER = 'x-vidtoolz-local-write-nonce';
```
Write endpoints check for this nonce header. Originates from the same server process.

### 3.4 State management
- **In-memory objects** for active jobs: `PRESTO_STATE`, `FLUX_STATE` (separate per-batch-runner).
- **Package Runs artifacts** read/written from a package-runs directory tree (per-run markdown/JSON files).
- **Episode Factory data** lives in browser `localStorage`, NOT on server. The server only serves static files and handles Package Engine thumbnail generation.
- **VIDNAS package state** scanned on each status API call (no caching, no indexing).

## 9. Areas Warranting Improvement Review

These are grounded observations, not recommendations. Each needs triage before action.

### 9.1 Monolithic server file (6,488 lines)
`package-engine-server.js` contains all routing, all endpoint handlers, all state management, and all artifact file logic in one file. The 47 endpoint constants suggest 47+ handler functions, plus helpers and middleware-equivalent request parsing.

Improvement angles:
- Split into route modules (e.g., `routes/aigen.js`, `routes/package-runs.js`, `routes/flux.js`)
- Extract artifact-file read/write logic into separate modules
- Extract job-manager (PRESTO_STATE, FLUX_STATE) into a reusable job-runner module
- Add a lightweight router (even if not Express) to reduce the if-chain

### 9.2 Two sub-systems in one repo
Episode Factory (browser-localStorage, v1.7.4) and Production Pipeline (Node server, aigen cockpit) share a repo but have different lifecycles, different data stores, different test strategies. Episode Factory is relatively mature and stable; Production Pipeline is the active development area.

Improvement angles:
- Clear directory separation (`/episode-factory/` vs `/cockpit/`)
- Separate README sections or separate top-level READMEs
- Consider whether they should be two repos (probably not, given the single-server reality, but the mental model should be explicit)

### 9.3 Untracked valuable work in git
Three new HTML pages + 6 new test files + 5 docs are untracked. Risk: accidental loss, no audit trail, no review path.

Improvement angles:
- Commit the new pages and their tests as a single changeset
- Move Codex prompts to a `prompts/` or `codex/` directory
- Add `.gitignore` rules for `.playwright-mcp/`, `inputs/`, `dogfood-output/` if they're runtime artifacts

### 9.4 Stale artifact directories
- `_smoke-topic-to-package-*` (3 folders) in `script-packages/`
- `xval-20260601`, `xval-pres`, `_xval_test2` in `batch-runs/`
- `Kling problem tausta`, `editors replaced Kling`, `Hermes kling`, `screengrab hermes to eka` — folder names with spaces and non-ASCII, suggesting ad-hoc debugging

Improvement angles:
- Archive policy: old smoke/validation runs go to a `stale-runs/` subdirectory
- Naming convention: require ASCII-safe, date-prefixed, lowercase-hyphenated
- Clean up folders with spaces — they'll break downstream scripts

### 9.5 No package schema
The `vidtoolz-youtube-ideas-20260611` folder has a rich but undocumented structure:
- What fields are required in `image-prompts.json`?
- What's the schema of `selected-images.json`?
- What's the contract between `flux-generation-manifest.json` and downstream tools?
- No schema file, no validation at the package level.

Improvement angles:
- A `package-schema.md` or JSON Schema file per artifact
- Validation at the API boundary (some exists in `POST /api/image-prompts/validate`)
- Version fields in each artifact (some have `version`, some don't)

### 9.6 Massive test files
- `package-run-scripts.test.js`: 460 KB
- `package-runs-dashboard.test.js`: 195 KB

Improvement angles:
- Investigate whether these are generated, heavily parameterized, or just sprawling
- Consider test grouping and selective running
- Check whether they have slow tests that should be tagged for exclusion from CI

### 9.7 No linting/formatting configured
No `eslint`, `prettier`, `standard`, `black`, `ruff`, or similar visible in the cockpit repo or aigen pipeline.

Improvement angles:
- Add ESLint with a sensible preset (e.g., `eslint:recommended`)
- Add `ruff` for Python
- Format on save, enforce in CI (if CI exists)

### 9.8 No CI/CD
No `.github/workflows/` beyond a PR template. No `npm test` automation, no `pytest` automation. Tests run manually by operator.

Improvement angles:
- GitHub Actions workflow for cockpit: `npm test`, `node --check` on every PR
- GitHub Actions for aigen: `pytest tests/`
- Or at minimum a pre-commit hook that runs syntax checks

### 9.9 Server uses no framework
Raw Node.js `http` module with manual routing. Works, but:
- No middleware chain
- No automatic JSON body parsing
- No standard error handling
- No route parameter extraction (manual `decodeURIComponent` + substring math)

Improvement angles:
- Consider Express, Fastify, or Hono — even if the team prefers "no deps"
- Or extract a thin routing/middleware layer in-repo

### 9.10 Nonce-based write protection is unusual
`LOCAL_WRITE_NONCE` mechanism adds a per-boot random nonce to write requests. This is a custom CSRF-like pattern. Questions to evaluate:
- What threat does it actually mitigate?
- Is it tested?
- Is it documented?
- Does it survive server restarts gracefully?

### 9.11 No backup or durability for VIDNAS state
VIDNAS is the canonical storage but has no backup, no git, no snapshots. Friction logs, manifests, decisions, and selected-images all live here with no versioning.

Improvement angles:
- `friction-log-*.md` files should be periodically mirrored to a git-tracked location
- Critical JSON artifacts (manifests, selections) could be commit-hook snapshotted to the cockpit repo
- Accept that media files are disposable; only metadata needs durability

### 9.12 Documentation gaps
- No architecture/README in aigen root describing the cockpit integration layer
- No documented API contract between cockpit and pipeline
- No "how to add a new API endpoint" guide
- No "how to add a new pipeline stage" guide

### 9.13 Image generation and selection state coupling
The chain `image-prompts.json → FLUX → selected-images.json → Wan` is well-wired but:
- No way to re-run FLUX for only modified prompts (full regeneration)
- `selected-images.json` uses a `version` field, but nothing enforces it matches current FLUX output
- No "this selection is stale relative to these images" warning in cockpit

### 9.14 Resolve assembly trigger is fire-and-forget
`POST /api/aigen/resolve-assembly/create` spawns the handoff generation script. The UI button disables when files exist. But:
- No diff between old and new handoff files shown
- No "what changed since last assembly?" visibility
- No rollback if the wrong files are generated

### 9.15 Two ComfyUI clients
- `image-to-video/diagnostics/run-wan22-diagnostic.py` has an inline, complete ComfyUI HTTP client
- `image-to-video/production/wan22-81f/run-production.py` has a different, more abstracted ComfyUI interaction

Improvement angles:
- Extract a shared ComfyUI client module
- Or explicitly document why they're different (diagnostic vs. production trade-offs)

## 10. What Is NOT Done (explicit boundaries)

These are intentional and should NOT be "improved" without evidence:

- **No cloud SaaS.** System is intentionally local-first. No multi-tenant billing, no AWS rendering.
- **No AI self-approval.** Human approves all durable state changes. Gates are explicit.
- **No automated topic/package generation.** Topic selection requires human judgment.
- **No Resolve automation.** Resolve timeline building is manual by Mikko's explicit choice.
- **No OBS automation.** PRESTO capture is manual.
- **No Finnish-language channel.** Concept evaluated and shelved. Do not revisit.
- **No backups for VIDNAS media.** Accepted risk. Selective manual archiving only.
- **No hidden automation.** The operator sees every state change.

## 14. Where to Start

If you had to pick one improvement target with the highest leverage:

**§9.1 — Split the monolithic server file.** Everything else (testability, readability, review, onboarding) improves when the 6,488-line file is broken into routed modules with clear ownership boundaries. The 47 endpoint constants already suggest the module seams.

If you want the lowest-friction win:

**§9.3 — Commit the untracked work.** The three new cockpit pages and their tests are valuable, completed, and at risk of accidental loss. One commit closes the loop.

If you want the highest-impact documentation fix:

**§9.5 — Document the package schema.** The `vidtoolz-youtube-ideas-*` layout is the most reused artifact contract in the system. A single schema markdown file prevents re-discovery on every new package.

If you want to reduce invisible state decay:

**§9.4 — Archive stale artifact directories.** The `_smoke-*` folders and `xval-*` runs are evidence of past validation. Archive them to a `stale-runs/` subdirectory so the active `script-packages/` list reflects real work.
